1.What is Node.js?
 Node.js is a js runtime environment that allow as to write and execute js code outside the browers. It is built in chrome's v8 engine, which compiles js into machine code, making it faster and effient.
 Pros->Fast Execution, Asychronous non-blocking.
 Cons->Single Threaded, Not suitable for CPU-intensive tasks.

7.Event Loop.
 Asychronous task are handled by event loop. It has five phases:
 -> Microtask(process.nexttick(higher priority ,promises))
 -> Timer callback(settimeout,setInterval)
 -> I/O callback (fs, reading or writing).
 -> Check queue (Setimmediate).
 -> Close queue (socket.on(),server.on()).

8.Libuv
 It is a C library in Node.js that implements/manages event loop. Thread Pool for blocking task (default 4 libuv threads).

10.Streams
 Streams let as to process large files in chunks small or units instead of loading them directly in memory at once. Handled by buffer class.

 2.Writable Streams -> For writing a big file.(fs.createWriteStream('input.txt').write('hello').end());
 3.Duplex Streams -> For reading and writing.
 4.Transform Streams -> To update streams while reading or writing.

    iii.Child Process -> used to create new process that run independently that execute external scripts, shell commands in node application.
    iv.Thread Pooling -> Internal pool used by libuv for running tasks in background. By default 4-threads.

37.Rate Limiting 
 Way to limit the number of request a user/client can make at a particular time.

    let rateLimiterStore = {};

    function rateLimiter(req,res,next){
        let ip = req.ip;
        let currentTime = Date.now();
        let windowTime = 60*1000;
        
        if(!rateLimiterStore[ip]){
            rateLimiterStore[ip] = [];
        }

        rateLimiterStore[ip] = rateLimiterStore[ip].filter((timestamp)=>currentTime-timestamp<windowTime);

        if(rateLimiterStore[ip].length>=9){
            return res.status(429).json({message:"Too many requests"});
        }
        rateLimiterStore[ip].push(currentTime);
        next();
    }

60.Polling Phase.
 It is a phase where node.js checks for completed i/o operations and executes their callbacks.
 ->The poll phase in the Node.js event loop is responsible for retrieving completed I/O operations and executing their callbacks.
 ->It is a way to handle completed I/O operations efficiently without blocking the main thread.

61.Long polling.
 It is a technique where the client send a request to server and server does not respond immediatly, server wait until new data is available. Then server sends response. Client immediatly sends the request again.
-->Client sends request → server
-->Server does NOT respond immediately
-->Server waits until:
 ->New data is available ✅
 ->OR timeout happens
-->Server sends response
-->Client immediately sends a new request again.

62.Server-Sent Events.
SSE-> It is a technique where the server continuiosly sends updates to the client over a single http connection.
   -> It is one-way server to client only.

63.Long Polling vs websockets vs Sever-Side-Events.
long polling -> protocol is http, connection type is continious requests, client->server,not real-time,low effiency, used in legacy systems(too many http requests,not effient for scale).
websockets -> protocol is websockets, connection type is persistant, two way client<->server, real-time , high effiency, used in chat,gaming(slightly complex to implement,needs websocket support in server).
sse-> protocol is http, persistant,one way server->client, realtime, medium effiency, used for notifications,feeds(client cannot send data back, limited browser support).

65.No cache middleware.
 It prevents the browser from storing(caching) responses.(when user logout and press back button still sees the previous page, it happens because of browser cache).
 -> it sets http headers like:
  Cache-Control: no-store, no-cache, must-revalidate, private
  Pragma: no-cache
  Expires: 0
  -> this tells the browser don't store this page. always fetch new data.
  ->At its core, no-cache middleware just sets HTTP response headers that instruct the browser not to store or reuse the response.

70.CSRF (cross-site request forgery).
 It is a attack where a malicious/harmfull website tricks a logged in users brower into sending a request to another site without users permission.
 Preventation methods - CSRF token,check origin, use post or put.

 When I visit a trusted site the browser stores the cookies when i visit a harmful website it makes a request to a trustable website without user permision with those cookies.

71.XSS (cross-site scripting).
 The attacker injects harmful js into a trustable website, that script runs on other users brower to steal data.
 Preventation methods - helmet,sanitize user input.

 The attacker tries to inject malicious/harmful JavaScript into a trusted webpage:
 ->Website stores/displays that script
 ->Another user visits the page
 ->Browser executes the attacker’s script automatically
 ->The attacker can:
    steal cookies/session tokens
    redirect users
    make fake requests
    modify the page
    act as the logged-in user.

81.Router Chaining.
1. Cleaner Code

Keeps all operations related to the same route together.

2. Better Readability

Easy to understand which methods belong to a particular endpoint.

3. Reduces Repetition
