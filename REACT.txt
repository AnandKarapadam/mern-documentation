1.What is React?
 React is a js library used to build user interface very efficiently and fast.
 Created by facebook meta.
 Limitation -> Needs extrenal libraries, like react-router, state management, http request.

2.Library vs Framework
Library -> It is group of functions or tools we use it when we need it. More flexible and lightweight.(React,jqery).
Framework -> It provides a complete structure from building an application. Heavy weight.(Angular,Django,Spring). 

3.Vite
 It is a modern frontend build tool and development server used to create very fast web apps.
 - Build Tool -> It converts our source code into production ready code. It bundles our code (does not use babel by default) it uses EsBuild instead of that. 

4.Vite vs CRA
 Vite -> Use native es modules. Supports TS,jsx, environment variable.
 CRA -> Bundles with webpack. Supports TS,jsx,environment variables but need external configuration.

5.HMR (Hot Module Replacement).
 On development when i change a file in the project it only update the part of the file it does not reload the entire page.

6.DOM VS Virtual DOM
DOM -> Virtual representation of HTML document as a tree.Every node represents every element.
Virtual DOM -> It is a copy of DOM used for Reconciliation, It will create a copy when component state changes.

7.SPA vs MPA
Single Page Application -> Loads on one HTML page only once js updates the content dynamicaly, faster navigation , initial loading can be slower.
Muliple Page Application -> Loads a new HTML page from the server every time when we click on it. Slower navigation, initial loading very fast.

8.Client Side Rendering vs Server Side Rendering.
CSR -> Render happens in the brower using js(react,vue,Angular).(REACT).  
SSR -> Rendering happens on the server brower recieves fully build HTML.(NEXT.js,EJS).

9.Default Exports vs Named Exports.
Default Exports -> Used to export one main component in file, only one per file possible.
Named Exports -> Can be many in a file, i have to use the exact name while importing in another file.

 First it take a copy of DOM tree and when state changes it take another copy and compare it and only changes the updated elements instead of recreating the entire Dom elements(diffing algorithm).

41.React.memo
 It prevent re-render of components unless the props change.
 let About = React.memo((props)=>{

 })
 When parent component re-renders the memoized child component performs shallow comparison with the previous props and the new props. It will only rerender when new props changed.

 If any prop is different, React re-renders the component.

45.useReducer().
 Alternative as useState() it is used when state logic become complex.

 function reducer(state,action){
    Switch (action.type){
        case "inc":
        return state+1;
        case "dec":
        return state-1;

        default:
        return state
    }
 }

 let app = ()=>{
    let [count,dispatch] = useReducer(reducer,0);

    return (
        <h1>{count}</h1>
        <button onClick={()=>dispatch({type:'inc'})}>+</button>
    )
 }

56.Concurrent Rendering/Mode.
 Normaly react renders where synchronous, with concurrent mode react can pause,resume or cancel rendering. Prioritize important updates(typing, clicking).
 Earlier it was called concurrent mode but now it is built in feature, enable by using:
 i.setTransition->Marks updates as low priority.
 ii.useTransition->Help show loading state.
 iii.useDeferredValue->Delays updates.
 Concurrent React allows React to work on multiple updates without blocking the UI, by prioritizing and interrupting rendering.

 ii.Forcing component reset.
  <UserForm key={userId} />
  When users changes, react unmounts old components creates a freash new one.

 ii.Switching between components.

  : <Form key="signup" />
 }

61.Lazy initilization of state.
 The initial state function runs only once(on first render) instead of every render.
 const [value, setValue] = useState(expensiveFunction());->runs on every render.
 const [value, setValue] = useState(() => expensiveFunction());->only runs once.
 use case:performance,memory efficiency,faster initial load.

62.useTransition().[introduced in react 18].
 Used for slow updates without blocking the UI. React mark this update as slow priority.
 -> let us to tell react which react updates are low priority.

 import { useTransition, useState } from "react";

const [search, setSearch] = useState("");
const [filtered, setFiltered] = useState(products);

const [isPending, startTransition] = useTransition();

function handleChange(e) {
  const value = e.target.value;

  // 🔥 urgent (instant)
  setSearch(value);

  // 🐢 non-urgent (heavy work)
  startTransition(() => {
    const result = products.filter(p =>
      p.name.toLowerCase().includes(value.toLowerCase())
    );
    setFiltered(result);
  });
}

63.useDifferedValue().(react 18).
 It is a React hook that returns a delayed version of a value, allowing heavy updates based on that value to happen later so the UI stays fast and responsive.
 ->It just delays when a value updates, which indirectly can reduce heavy rendering impact.
 ->useDeferredValue holds the previous value for a short time and updates to the new value later.

 import { useDeferredValue, useState } from "react";

const [search, setSearch] = useState("");

// 👇 delayed version
const deferredSearch = useDeferredValue(search);

const filtered = products.filter(p =>
  p.name.toLowerCase().includes(deferredSearch.toLowerCase())
);

return (
  <>
    <input value={search} onChange={e => setSearch(e.target.value)} />

    <ul>
      {filtered.map(p => <li key={p.id}>{p.name}</li>)}
    </ul>
  </>
);

65.SOLID Principle.
 S(Single Responsibility Principle SRP) -> Split components, use custom hooks. One component is for one job.
 O(Open Closed Principle OCP) -> Open for extension, Closed for modification. You should be able to add new functionality without changing existing code.
 L(Liskov Substitution Principle LSP) -> You should be able to replace a parent component with a child component without breaking the expected behaviour.
 I(Interface Segrigation Principle ISP) -> Don't force components to accept props they don't need.
 D(Dependency Inversion Principle DIP)(Dependency Injection) -> High-level components should not depend on low-level components. Don't hardcode dependencies.Hard to debug if error happens.
